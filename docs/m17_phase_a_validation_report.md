## TL;DR

| Evaluator | Gate | Outcome |
|---|---|---|
| **E1 — ASR (Whisper / ivrit-ai)** | median WER < 0.40 on Hebrew clips | **PASS** by ~10× margin |
| **E2 — UTMOS** | clean mean − degraded mean ≥ 0.5 | **FAIL** (separation = −0.02) |

- **E1 — GO.** Both Whisper variants are dramatically more accurate on our Hebrew TTS output than the gate assumes. Adopt `ivrit-ai/whisper-large-v3` as primary; keep `openai/whisper-large-v3` as a hallucination-check baseline.
- **E2 — NO-GO under the current design.** UTMOS does not discriminate clean from degraded on our data. Phase A's E2 module must not be implemented as designed; re-spike with turn-segmented UTMOS before committing, and consider dropping predicted MOS in favour of objective signal metrics + human listening (the M12 incident protocol).

## Findings

### 1. ASR is far better than the gate assumed

The 0.40 WER gate was conservative-by-design (Whisper has well-known weaknesses on low-resource languages, and Hebrew was a wager). On our actual data both models clear the gate by ~10×. Implication for E1 design: clip-level pass thresholds can be set in the 0.10–0.15 WER range (3–5× the typical observed value) and still allow plenty of headroom. Calibration per `automated_eval_design.md` lines 400–410 still required before any gate is enforced — these numbers are the floor, not the target.

### 2. ivrit-ai/whisper-large-v3 beats openai/whisper-large-v3 on Hebrew

Median WER drops 37% (0.054 → 0.034) and the win is consistent across 7 of 8 clips. Adopt as primary E1 model.

### 3. ivrit-ai has a hallucination failure mode that openai does not

On `sp_it_a_0002_00`, ivrit-ai produced a hypothesis that **re-transcribed a chunk of the conversation a second time** with `U+202B` (Right-to-Left Embedding) prefixes — a known Whisper "repeated content" loop. WER tripled (0.065 → 0.144) and CER 4×'d (0.026 → 0.094) compared to openai on the same clip. The hyp_words count (305) exceeds the ref_words count (291) by 4.8% — a useful guard.

**Mitigations to bake into E1:**
- **Length sanity check:** flag any clip where `hyp_words / ref_words > 1.05` (or equivalent on character count). Trigger reprocessing.
- **Cross-check on outliers:** when ivrit-ai produces an anomalously high WER on a clip with a clean reference, re-run with openai/whisper-large-v3. If openai is materially lower, prefer openai for that clip.
- **Repeat-n-gram detection:** strip-and-count tri-gram repetitions in the hypothesis; high repetition without matching reference repetition indicates a loop.

The `pipeline(chunk_length_s=30)` codepath is documented as "very experimental" by HF for seq2seq; long-term we should switch E1 to `WhisperForConditionalGeneration.generate(...)` directly, which uses Whisper's native long-form algorithm (and is what the original Whisper paper §3.8 specifies). That switch is its own follow-up.

### 4. UTMOS is not fit for purpose on this data

The gate fail is not borderline — the *direction* is wrong. Two compounding causes:

- **Distribution mismatch.** UTMOS22 was trained on short utterances (typically 4–10 s) from English/Japanese MOS subjective datasets. Our clips are 99–156 s of multi-turn Hebrew dialogue with ≥0.5 s silence padding head/tail and gaps between turns. The model's pooling does not generalise: every clean clip clusters tightly between 1.28 and 1.34 — a 0.06 spread on a 1–5 scale. That's a model floor effect, not a quality signal.
- **−10 dB SNR white noise pushes the prediction *up* slightly.** Plausible explanation: noise masks the artefacts UTMOS is most sensitive to (silence regions, plosive clicks), or the noise floor matches what UTMOS reads as "more natural recording conditions" relative to the very-clean TTS output. Either way: UTMOS sees the noise as *quality-positive*, which is incompatible with using it as a degradation detector.

### 5. The design doc names a model that does not exist

`docs/automated_eval_design.md:421` references `ivrit-ai/whisper-v2-d3-e3`. That identifier does not resolve on HuggingFace; the canonical Hebrew-specialised ivrit-ai model is `ivrit-ai/whisper-large-v3`. The design doc should be updated. (`whisper-large-v3-ct2` and `whisper-large-v3-turbo-ct2` also exist for CTranslate2 / faster-whisper deployment.)

## Limitations

1. **Sample size: 8 clips, not 10.** The eight clips span four typologies (IT/NEG/NEU/SV × two scenes each). For the ASR gate, 8 vs 10 is robust — both models pass by ~10× margin. For UTMOS the spread is so narrow that 2 more clips would not change the failed direction.
2. **Single speaker pair.** All clips are `AGG_M_30-45_001` (he-IL-AvriNeural) + `VIC_F_25-40_002` (he-IL-HilaNeural). Speaker generalisation is not tested.
3. **Tier A only.** No room IR, no device profiles, no background noise. Tier B/C ASR performance is unknown — likely materially worse and warrants its own validation before E1 gates production data.
4. **Reference normalisation.** Hebrew niqqud (cantillation/diacritics) is stripped from reference and hypothesis before WER scoring; Whisper outputs unpointed text. Punctuation is normalised to whitespace. No casing — Hebrew has no case.
5. **Long-form chunking.** Transcription uses HF's 30 s sliding-window via `pipeline(chunk_length_s=30)`, which HF flags as experimental for seq2seq. Switching to the native `generate(...)` long-form algorithm is a follow-up.
6. **No CI cost measurement.** The script captures wall time, but the laptop closed mid-run; the numbers are not measurements of inference cost. A controlled benchmarking session (wall power, lid open, no sleep) is required before quoting per-clip latency in the design doc's CI cost table (`automated_eval_design.md` lines 437, 449–456).

## Recommendation

### E1 — Whisper ASR — GO

1. **Adopt `ivrit-ai/whisper-large-v3` as the primary E1 model.** Keep `openai/whisper-large-v3` available as a fallback for outlier reprocessing.
2. **Update the design doc** at `docs/automated_eval_design.md:421` to replace `ivrit-ai/whisper-v2-d3-e3` with `ivrit-ai/whisper-large-v3`.
3. **Bake hallucination guards into E1** before enabling any gate based on WER:
   - Length-ratio sanity check (`hyp_words / ref_words > 1.05` → flag).
   - Repeated-tri-gram detector on the hypothesis.
   - Cross-check anomalies with openai/whisper-large-v3 and prefer the lower of the two when they disagree.
4. **Defer the long-form chunking switch** (HF `pipeline(chunk_length_s=...)` → native `generate(...)`) to a focused follow-up — it doesn't block Phase A but should land before E1 is wired into release gates.
5. **Calibrate E1 thresholds** per the design doc's calibration protocol on a 50-clip baseline before any WER threshold gates merges.

### E2 — Predicted MOS — NO-GO under current design

UTMOS22 as a single-shot evaluator on long-form Hebrew TTS is broken for our purpose. Three options, in increasing order of cost:

- **Option A (cheapest, do first):** turn-segment the audio using the `ONSET`/`OFFSET` already in the reference transcripts, run UTMOS at the turn level (4–15 s windows match its training distribution), and aggregate per clip. Re-spike: if turn-level UTMOS shows ≥0.5 separation between clean and degraded, E2 design proceeds with turn-segmented scoring. If still no separation, escalate to Option B/C.
- **Option B:** swap UTMOS for a long-form-friendly MOS predictor — `NISQA` (full-band, supports arbitrary windows, has been used for synthesis quality) or `WV-MOS`. Re-spike on the same 8-clip set + 5 degraded.
- **Option C (no automated MOS):** drop predicted MOS from the eval pipeline. Replace E2 with deterministic objective metrics (SNR estimate, peak dBFS, clipping rate, silence ratio, spectral tilt) and rely on the human listening-test cadence already established by the M12 incident. This is what the May 2026 M12 listening test effectively concluded — automated MOS could not catch the breathiness regression that human ears caught immediately.

Recommendation: **start with Option A** (one afternoon's work; same script, segment-and-aggregate). If A fails, jump straight to Option C — the M12 history says automated MOS is unreliable as a release gate for this project, and Option B is unlikely to cross the same gap.

### Phase A overall

Phase A is **partially unblocked**:
- E1 code skeleton can proceed.
- E2 needs a follow-up spike (Option A above) before any E2 implementation begins.

The Phase A code-skeleton PR should land E1 only; E2 follows once the re-spike confirms a viable approach.
